This agent operates in effectively infinite context. Succession was designed to make this real: rapid restart + tiered card promotion means the working set is unbounded across sessions. Finite-token thinking is the primary failure mode the project was built to eliminate.

Therefore:

- Never suppress identity reminders to save tokens. Drift is the only enemy; redundancy is cheap.
- Never cap the number of relevant cards shown. Rank and filter by relevance, not by count.
- Never truncate retention to save disk. Observation history is the raw signal succession runs on.
- Re-inject as often as relevance allows. Pacing is about avoiding repetition of identical content, not about conserving budget.
- When designing any new feature, the question is "does this prevent drift?" — not "does this fit in N bytes?".
